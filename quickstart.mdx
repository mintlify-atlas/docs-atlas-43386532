This guide walks you through installing `sema`, running the canonical example from the README, and using `Len` and `IsEmpty` to inspect semaphore state at runtime.

<Steps>
  <Step title="Install sema">
    Add the package to your Go module:

    ```bash
    go get -u github.com/andreimerlescu/sema
    ```

    `go get` downloads the module, adds it to your `go.mod`, and pins the version in `go.sum`. Because `sema` has no external dependencies, nothing else is added to your module graph.

    <Note>
      `sema` requires Go 1.21 or later. Run `go version` to check your installed version.
    </Note>
  </Step>

  <Step title="Import the package">
    Add the import path to your Go source file alongside the standard-library packages you'll need:

    ```go
    import (
        "fmt"
        "sync"
        "sync/atomic"
        "time"

        "github.com/andreimerlescu/sema"
    )
    ```
  </Step>

  <Step title="Run the complete example">
    The following program is the canonical example from the README, also verified as `TestReadMeExample` in the test suite. Paste it into `main.go` and run it with `go run main.go`.

    ```go
    package main

    import (
        "fmt"
        "sync"
        "sync/atomic"
        "time"

        "github.com/andreimerlescu/sema"
    )

    func main() {
        workers := 10 // 10 workers
        mySemaphore := sema.New(workers)
        var wg sync.WaitGroup
        var delaySeconds atomic.Int32
        for i := 0; i < workers*2; i++ {
            wg.Add(1)
            mySemaphore.Acquire()
            go func(i int, wg *sync.WaitGroup) {
                defer wg.Done()
                defer mySemaphore.Release()
                delay := delaySeconds.Add(1)
                time.Sleep(time.Duration(delay) * time.Second)
                fmt.Printf("worker %d finished after %d seconds\n", i, delaySeconds.Load())
            }(i, &wg)
        }
        wg.Wait()
        fmt.Printf("wait group released with %d workers left in the semaphore\n", mySemaphore.Len())
    }
    ```

    The loop schedules `workers*2` (20) goroutines, but only `workers` (10) run at any given time. The remaining 10 block on `Acquire` until a running goroutine calls `Release`.
  </Step>

  <Step title="Understand each part">
    Here is what each section of the example does:

    **Create the semaphore**

    ```go
    workers := 10
    mySemaphore := sema.New(workers)
    ```

    `sema.New(10)` allocates a buffered channel with capacity 10. At most 10 goroutines can hold a slot at the same time.

    **Acquire before spawning the goroutine**

    ```go
    mySemaphore.Acquire()
    go func(i int, wg *sync.WaitGroup) { ... }(i, &wg)
    ```

    `Acquire` is called on the main goroutine, *before* `go func`. This is intentional: the slot is claimed synchronously, so the loop itself blocks when all 10 slots are taken. No goroutine is ever spawned without a slot already reserved — preventing unbounded goroutine creation.

    <Tip>
      Always call `Acquire` before `go func`, not inside it. Calling it inside
      the goroutine still bounds concurrent work, but lets all goroutines be
      spawned first, which consumes memory proportional to the total number of
      items before any throttling takes effect.
    </Tip>

    **Release via defer**

    ```go
    defer wg.Done()
    defer mySemaphore.Release()
    ```

    Both defers fire when the goroutine returns, even if it panics. `defer` ensures `Release` is never skipped, avoiding a deadlock where the semaphore fills up and no one ever drains it.

    <Note>
      `defer` statements execute in LIFO order. Here `wg.Done()` is deferred
      first and `mySemaphore.Release()` second, so `Release` runs before
      `wg.Done()`. This means the semaphore slot is freed before the WaitGroup
      counter decrements — which is safe and correct.
    </Note>

    **Wait for all goroutines to finish**

    ```go
    wg.Wait()
    ```

    `sync.WaitGroup` tracks outstanding goroutines. `wg.Add(1)` increments the counter before each goroutine starts; `defer wg.Done()` decrements it when the goroutine returns. `wg.Wait()` blocks the main goroutine until the counter reaches zero.

    `sema` and `sync.WaitGroup` serve different purposes:

    | | `sema.Semaphore` | `sync.WaitGroup` |
    |--|--|--|
    | **Purpose** | Limit concurrent goroutines | Wait for all goroutines to finish |
    | **Blocks when** | Capacity is full | Counter is non-zero |
    | **Releases when** | `Release()` is called | `Done()` is called |
  </Step>

  <Step title="Inspect semaphore state">
    Use `Len` and `IsEmpty` to observe how many slots are occupied at any point:

    ```go
    sem := sema.New(3)

    sem.Acquire()
    fmt.Println(sem.Len())     // 1
    fmt.Println(sem.IsEmpty()) // false

    sem.Acquire()
    fmt.Println(sem.Len())     // 2

    sem.Release()
    sem.Release()
    fmt.Println(sem.IsEmpty()) // true
    ```

    After `wg.Wait()` returns in the full example, all goroutines have finished and released their slots, so `mySemaphore.Len()` returns `0` and `mySemaphore.IsEmpty()` returns `true`.

    ```go
    wg.Wait()
    fmt.Printf("wait group released with %d workers left in the semaphore\n", mySemaphore.Len())
    // Output: wait group released with 0 workers left in the semaphore
    ```

    <Info>
      `Len()` reads `len(channel)`, which is an O(1) operation in Go and safe
      to call from any goroutine at any time.
    </Info>
  </Step>
</Steps>
